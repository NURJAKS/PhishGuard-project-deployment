# Документация: Кнопка "DNS проверка" в PhishGuard Extension

## Общее описание

Кнопка "DNS проверка" (DNS Check) - это функциональность браузерного расширения PhishGuard, которая выполняет комплексный анализ DNS-записей домена текущей открытой веб-страницы для выявления потенциальных признаков фишинга и мошенничества.

## Расположение в коде

### Основные файлы:
- **popup.js** (строки 505-767) - основная реализация для popup-окна расширения
- **admin-panel.js** (строки 496-770) - аналогичная реализация для админ-панели
- **popup.html** (строки 233, 272-287) - HTML разметка UI элементов
- **admin-panel.html** (строки 111, 116-126) - HTML разметка для админ-панели
- **manifest.json** (строки 20-22, 28) - разрешения и CSP настройки

## Используемые инструменты и API

### 1. DNS-over-HTTPS (DoH) провайдеры

#### Cloudflare DNS (основной)
- **URL**: `https://cloudflare-dns.com/dns-query`
- **Метод**: GET
- **Параметры**: 
  - `name` - доменное имя (URL-encoded)
  - `type` - тип DNS записи (A, AAAA, MX)
- **Headers**: `Accept: application/dns-json`
- **Формат ответа**: JSON (DNS-over-HTTPS JSON format)
- **Функция**: `queryCloudflare(domain, type)`

#### Google DNS (резервный/fallback)
- **URL**: `https://dns.google/resolve`
- **Метод**: GET
- **Параметры**: 
  - `name` - доменное имя (URL-encoded)
  - `type` - тип DNS записи (A, AAAA, MX)
- **Формат ответа**: JSON
- **Функция**: `queryGoogleDns(domain, type)`
- **Использование**: Используется как fallback, если Cloudflare DNS не отвечает или возвращает ошибку

### 2. IP Geolocation API

#### ipinfo.io
- **URL**: `https://ipinfo.io/{ip}/json`
- **Метод**: GET
- **Параметры**: IP-адрес в пути URL
- **Формат ответа**: JSON
- **Возвращаемые данные**:
  - `city` - город
  - `region` - регион/область
  - `country` - код страны (ISO 3166-1 alpha-2)
  - `loc` - координаты (широта, долгота)
  - `org` - организация/провайдер (ASN информация)
  - `hostname` - hostname IP-адреса
  - `hosting` - boolean, является ли IP дата-центром
- **Функция**: `getIpInfo(ip)`

## Алгоритм работы

### Шаг 1: Валидация и подготовка
1. Проверка наличия активной вкладки (`currentTab`)
2. Проверка, что URL является анализируемым (`isAnalyzableUrl()`)
   - Должен быть HTTP или HTTPS протокол
   - Не должен быть внутренней страницей браузера (chrome://, extension://)
3. Извлечение домена из URL: `new URL(currentTab.url).hostname`
4. Инициализация UI элементов (показ панели, сброс предыдущих данных)

### Шаг 2: DNS запросы (параллельные)
Выполняются три типа DNS-запросов:

1. **A записи (IPv4)**
   - Тип: `A`
   - Цель: Получить IPv4 адреса домена
   - Обработка: Фильтрация записей с `type === 1`, извлечение `data` поля

2. **AAAA записи (IPv6)**
   - Тип: `AAAA`
   - Цель: Получить IPv6 адреса домена
   - Обработка: Фильтрация записей с `type === 28`, извлечение `data` поля

3. **MX записи (Mail Exchange)**
   - Тип: `MX`
   - Цель: Получить почтовые серверы домена
   - Обработка: Фильтрация записей с `type === 15`, парсинг формата `priority server`

### Шаг 3: Обработка DNS результатов
- Объединение всех IP-адресов: `[...ipv4List, ...ipv6List]`
- Форматирование IP для отображения (разделение IPv4/IPv6)
- Парсинг MX записей: разделение приоритета и сервера
- Отображение результатов в UI

### Шаг 4: Геолокация и информация о провайдере
- Выбор первого IPv4 адреса из списка
- Запрос к ipinfo.io для получения:
  - Географического местоположения (город, регион, страна)
  - Информации о провайдере/организации (ASN)
  - Hostname сервера
  - Признаков дата-центра

### Шаг 5: Анализ рисков
Функция `analyzeRisk(ipInfo, domain)` выполняет:

#### Проверка типа провайдера:

**Известные хостинг-провайдеры** (KNOWN_HOSTING_PROVIDERS):
- amazon, aws, digitalocean, linode, vultr, ovh, hetzner
- google, gcp, azure, microsoft, cloudflare, fastly, akamai
- godaddy, hostgator, bluehost, namecheap, dreamhost, hostinger
- ionos, contabo, scaleway, upcloud, kamatera, rackspace
- **Результат**: Положительный признак (профессиональный хостинг)

**Известные ISP провайдеры** (KNOWN_ISP_PROVIDERS):
- comcast, verizon, at&t, spectrum, cox, centurylink
- rostelecom, beeline, megafon, mts, tele2, yota
- kazakhtelecom, kcell, activ, altel, tele2.kz
- **Результат**: +30 к riskScore (подозрительный признак - домашний провайдер)

#### Проверка VPN/Proxy:
- Поиск в `org` или `hostname`: "vpn", "proxy", "tunnel"
- **Результат**: +20 к riskScore

#### Проверка страны (только в admin-panel.js):
- Подозрительные страны: RU, CN, KP, IR, BY
- **Результат**: Информационное сообщение

#### Проверка hostname:
- Сравнение hostname сервера с доменом
- Если не совпадает - информационное сообщение

#### Проверка дата-центра:
- Использование поля `ipInfo.hosting === true`
- **Результат**: Положительный признак

### Шаг 6: Определение уровня риска
На основе `riskScore`:
- **low** (0-19): Низкий риск
- **medium** (20-39): Средний риск
- **high** (40+): Высокий риск

### Шаг 7: Формирование результата
Структура ответа `analyzeRisk()`:
```javascript
{
  level: 'low' | 'medium' | 'high',
  badge: '🟢 Низкий риск' | '🟡 Средний риск' | '🔴 Высокий риск',
  summary: 'Текстовое описание',
  details: ['Массив деталей анализа'],
  isHosting: boolean,
  isResidential: boolean
}
```

### Шаг 8: Обновление UI
Отображаются следующие элементы:
- **Статус**: Индикатор выполнения (точка + текст)
- **Домен**: Проверяемый домен
- **IP адреса**: Список всех IPv4 и IPv6 адресов с метками
- **MX записи**: Почтовые серверы с приоритетами (если есть)
- **Геолокация**: Город, регион, страна, координаты
- **Провайдер**: Организация/ASN информация
- **Риск-бейдж**: Цветной индикатор уровня риска
- **Детали риска**: Развернутое описание найденных признаков

## Структура данных DNS ответа

### Формат ответа Cloudflare/Google DNS:
```json
{
  "Status": 0,  // 0 = успех
  "Answer": [
    {
      "name": "example.com",
      "type": 1,  // 1 = A, 28 = AAAA, 15 = MX
      "TTL": 300,
      "data": "192.0.2.1"  // IP для A/AAAA, "10 mail.example.com" для MX
    }
  ]
}
```

### Типы DNS записей:
- **type 1**: A запись (IPv4)
- **type 28**: AAAA запись (IPv6)
- **type 15**: MX запись (Mail Exchange)

## Обработка ошибок

### Уровни обработки:
1. **DNS запросы**: При ошибке Cloudflare → fallback на Google DNS
2. **IP информация**: При ошибке ipinfo.io → `ipInfo = null`, анализ продолжается
3. **Общие ошибки**: Try-catch блок с отображением сообщения об ошибке в UI

### Сообщения об ошибках:
- "DNS проверка доступна только для HTTP/HTTPS страниц"
- "DNS проверка недоступна для внутренних страниц браузера"
- "Не удалось выполнить DNS проверку: {error.message}"

## Особенности реализации

### Fallback механизм:
```javascript
async function queryDns(domain, type = 'A') {
    let result = await queryCloudflare(domain, type);
    if (!result || result.Status !== 0) {
        result = await queryGoogleDns(domain, type);
    }
    return result;
}
```

### Форматирование IP:
- IPv4: `IPv4: 192.0.2.1`
- IPv6: `IPv6: 2001:db8::1`
- Визуальное оформление: теги с моноширинным шрифтом

### Форматирование MX:
- Формат: `[priority] server`
- Пример: `[10] mail.example.com`

## Использование в контексте безопасности

Кнопка DNS проверки является частью системы защиты от фишинга и позволяет:
1. Проверить подлинность домена через DNS записи
2. Выявить использование подозрительных хостинг-провайдеров
3. Обнаружить признаки VPN/Proxy серверов
4. Определить географическое расположение сервера
5. Оценить общий уровень риска на основе инфраструктуры

## Технические детали

- **Язык**: JavaScript (ES6+)
- **API**: Fetch API для HTTP запросов
- **Формат данных**: JSON
- **Асинхронность**: async/await
- **Обработка ошибок**: Try-catch блоки
- **UI обновления**: Прямая манипуляция DOM элементами

## Зависимости

- Chrome Extensions API (chrome.tabs)
- Fetch API (встроенный в браузер)
- DNS-over-HTTPS протокол
- ipinfo.io API (бесплатный tier)

## Ограничения

1. ipinfo.io имеет лимиты на бесплатном тарифе
2. DNS запросы могут быть медленными при использовании fallback
3. Геолокация доступна только для IPv4 адресов
4. Анализ рисков основан на эвристиках, не является абсолютно точным
